# 03 - 工作流篇：AI 融入日常开发

> 五种核心开发场景下的 AI 使用模式：TDD、Code Review、调试、重构、文档。

## TDD with AI

**何时使用**：开发新功能、修复 bug 前先写测试。

**步骤**：

1. 描述功能需求，让 AI 先生成测试用例
2. 运行测试，确认全部失败（红灯）
3. 让 AI 编写实现代码
4. 运行测试，确认全部通过（绿灯）
5. 让 AI 重构代码，保持测试通过

```bash
# Claude Code 示例
> 为一个 calculateDiscount(price, tier) 函数写测试：
  - 普通用户 tier="standard" 无折扣
  - VIP 用户 tier="vip" 打 9 折
  - 价格为负数时抛出错误
  - 价格为 0 返回 0
  使用 Jest，先只写测试文件。

> 测试全部失败了，现在写实现代码让测试通过。
```

**常见陷阱**：

- 不要让 AI 同时写测试和实现——它会让两者互相"适配"而非真正测试功能
- 审查 AI 生成的测试是否覆盖了边界条件

## Code Review with AI

**何时使用**：PR 提交前自查、审查他人代码、学习不熟悉的代码。

**步骤**：

1. 将代码变更提供给 AI
2. 要求它从特定角度审查（安全、性能、可读性等）
3. 评估 AI 的建议，采纳合理的部分

```bash
# Claude Code 示例
> 审查 git diff HEAD~1 的变更，关注：
  1. 潜在的安全问题
  2. 性能瓶颈
  3. 错误处理是否完善
  4. 是否有更简洁的写法
```

**常见陷阱**：

- AI 可能过度优化——不是每个建议都值得采纳
- AI 不了解你的业务上下文，涉及业务逻辑的判断需要你自己做

## 调试 with AI

**何时使用**：遇到难以定位的 bug、看不懂的错误信息。

**步骤**：

1. 提供完整的错误信息（stack trace）
2. 描述期望行为和实际行为
3. 提供相关代码
4. 让 AI 分析原因并给出修复方案

```bash
# 好的调试提示词
> 运行 npm test 时出现以下错误：
  TypeError: Cannot read properties of undefined (reading 'map')
  at UserList (src/components/UserList.tsx:15:22)

  期望：组件正常渲染用户列表
  实际：首次加载时崩溃

  相关代码在 src/components/UserList.tsx
  数据来源于 useQuery hook
```

**常见陷阱**：

- 只贴错误信息不贴代码，AI 只能猜测
- 不说明已经尝试过的方案，AI 可能重复建议相同的思路

## 重构 with AI

**何时使用**：代码可以工作但需要改善结构、降低复杂度、提升可维护性。

**步骤**：

1. 明确重构目标（不改变外部行为）
2. 确保有测试覆盖
3. 让 AI 分析当前代码的问题
4. 逐步重构，每步验证测试通过

```bash
# Claude Code 示例
> 这个文件有 500 行，做了太多事情。请分析它的职责，
  建议如何拆分为多个模块。先给出方案，不要直接改代码。

（审查方案后）

> 方案 OK。先把数据库相关的逻辑提取到 src/repositories/user.ts，
  保持原有导出接口不变。
```

**常见陷阱**：

- 没有测试就重构——AI 重构后可能引入 bug，没测试就发现不了
- 一次重构太多——小步前进，每步可验证

## 文档 with AI

**何时使用**：API 文档、README、代码注释、架构文档。

**步骤**：

1. 指定文档类型和目标读者
2. 提供代码或已有文档作为输入
3. 指定格式要求（Markdown、JSDoc、OpenAPI 等）

```bash
# Claude Code 示例
> 阅读 src/api/ 目录下所有路由文件，
  生成 OpenAPI 3.0 格式的 API 文档。
  包含每个端点的描述、参数、请求体和响应示例。

# 代码注释
> 为 src/utils/crypto.ts 中的所有 public 函数添加 JSDoc 注释，
  包含参数说明、返回值和使用示例。
```

**常见陷阱**：

- AI 可能编造不存在的 API 参数——务必对照代码验证
- 不指定格式会得到不一致的输出风格
