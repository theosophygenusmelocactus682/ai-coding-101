# 06 - 团队协作篇：在团队中推广 AI 编程

> 从个人使用到团队标准化，覆盖引入策略、共享配置、安全与成本管理。

## 团队引入 AI 编程工具

### 分阶段推进

**第 1 阶段（1-2 周）：试点**

- 选 2-3 名有兴趣的开发者先行试用
- 选择一个中等复杂度的真实任务作为试点
- 记录效率提升数据和遇到的问题

**第 2 阶段（2-4 周）：扩展**

- 将试点经验整理为团队内部指南
- 举办一次内部分享会，展示实际案例
- 为项目创建标准化的配置文件（CLAUDE.md 等）

**第 3 阶段（持续）：标准化**

- 将 AI 工具纳入团队工作流标准
- 建立提示词库和 Skills 库
- 定期回顾和优化使用方式

### 应对常见阻力

| 阻力 | 应对方式 |
|------|----------|
| "AI 会取代我们" | AI 是效率工具，不是替代品。用它处理重复工作，你专注于创造性工作 |
| "AI 代码质量不靠谱" | 设定 Code Review 标准，AI 生成的代码和人写的代码使用相同的审查流程 |
| "学习成本太高" | 从简单任务开始（代码补全、写测试），逐步深入 |

## 共享配置和标准

### 项目配置文件模板

为团队创建标准化的配置文件模板，存放在公司内部仓库：

```
team-ai-config/
├── templates/
│   ├── CLAUDE.md.template     # Claude Code 项目配置模板
│   ├── cursorrules.template   # Cursor 配置模板
│   └── AGENTS.md.template     # Codex 配置模板
├── skills/
│   ├── code-review.md         # 代码审查 Skill
│   ├── pr-description.md      # PR 描述生成 Skill
│   └── api-endpoint.md        # API 端点创建 Skill
└── prompts/
    ├── bug-fix.md             # Bug 修复提示词模板
    └── feature.md             # 新功能提示词模板
```

### 统一代码规范声明

在配置文件中明确声明团队规范，确保每个成员的 AI 输出风格一致：

```markdown
## 团队代码规范
- 变量命名：camelCase（JS/TS）、snake_case（Python）
- 函数长度不超过 50 行
- 每个 public 函数必须有文档注释
- 错误处理：不允许空 catch 块
- 提交信息格式：type(scope): description
```

## AI 辅助团队的 Code Review

### 新的 Review 检查点

当团队使用 AI 工具后，Code Review 需要增加几个关注点：

- **理解度检查**：提交者是否理解 AI 生成的代码？能否解释关键逻辑？
- **过度工程检查**：AI 有时会过度设计。这个抽象层真的需要吗？
- **幻觉检查**：AI 引用的 API 或库函数是否真实存在？版本是否正确？
- **安全检查**：AI 是否引入了硬编码的密钥、不安全的依赖或注入漏洞？

### 推荐流程

```
开发者完成功能
    ↓
AI 自动 Review（第一轮）
    ↓
修复 AI 发现的问题
    ↓
人工 Review（第二轮）—— 关注业务逻辑和架构
    ↓
合并
```

## 安全注意事项

### 必须做的

- 确认 AI 工具的数据处理政策（是否用你的代码训练模型）
- 在 `.gitignore` 和 AI 配置中排除敏感文件（`.env`、密钥文件等）
- 审查 AI 生成的代码中的依赖项——确保使用的包真实存在且版本安全
- 对涉及认证、加密、支付的代码进行额外的人工审查

### 不要做的

- 不要将生产环境的 API 密钥粘贴到 AI 对话中
- 不要让 AI 直接操作生产数据库
- 不要在公共 AI 工具中讨论未公开的商业机密

## 成本管理

### 控制策略

| 策略 | 说明 |
|------|------|
| 设置用量配额 | 为每个成员设置月度 API 调用限额 |
| 选择合适的模型 | 简单任务用便宜的模型，复杂任务用高端模型 |
| 优化提示词 | 好的提示词减少反复尝试的次数 |
| 利用项目配置 | CLAUDE.md 等配置文件减少每次都要重复提供上下文 |
